# 项目讲解模板

适用场景：项目介绍、述职汇报、面试项目回答、临时说明、向非技术方解释方案。

原则：
- 先结论，后细节。
- 先讲问题和价值，再讲实现。
- 少讲“我写了什么代码”，多讲“为什么这样做、带来了什么结果”。

## 1 分钟版本

一句话结论：

`我做的是 [项目/模块]，主要解决 [核心问题]，最终把 [关键指标/结果] 从 [原始状态] 改善到 [目标状态]。`

展开：

1. 问题
`当时的主要问题是 [痛点/瓶颈/风险]，它直接影响了 [用户/业务/交付]。`

2. 方案
`我采用了 [核心方案]，重点做了 [2-3 个关键动作]。`

3. 价值
`结果是 [指标变化/稳定性提升/效率提升/成本下降]，这个方案的价值在于 [一句话总结]。`

## 3 分钟版本

### 问题

- 背景：`这个项目发生在 [业务场景/阶段]。`
- 痛点：`原来存在 [性能/稳定性/效率/协作/成本] 问题。`
- 影响：`它导致 [用户体验差/交付慢/成本高/风险大]。`

### 方案

- 我的目标：`我要解决的是 [核心目标]。`
- 我的判断：`我没有选 [备选方案]，因为 [原因]；最终选择 [当前方案]，因为 [原因]。`
- 关键动作：
  `第一，我做了 [动作 1]。`
  `第二，我做了 [动作 2]。`
  `第三，我做了 [动作 3]。`

### 价值

- 结果：`最终 [指标/现象] 从 [A] 变成了 [B]。`
- 验证：`我通过 [监控/压测/线上反馈/交付结果] 验证方案有效。`
- 复用性：`这套做法后续还能复用在 [别的模块/别的项目]。`

## STAR 版本

- Situation
  `当时处于 [业务背景]，系统/团队面临 [具体问题]。`
- Task
  `我的任务是在 [时间/资源约束] 下完成 [目标]。`
- Action
  `我做了 [关键分析]、[关键设计]、[关键落地动作]。`
- Result
  `最后得到 [量化结果]，并且额外带来 [稳定性/效率/协作收益]。`

## PREP 版本

- Point
  `我认为这个项目最关键的点是 [结论]。`
- Reason
  `原因是 [1]、[2]、[3]。`
- Example
  `例如我在项目里通过 [动作]，解决了 [问题]。`
- Point
  `所以这件事的核心不是 [表面动作]，而是 [真正价值]。`

## 技术人最容易讲偏的地方

- 不要一上来讲实现细节，先讲问题和结论。
- 不要只说“用了什么技术”，要说“为什么选它，不选别的”。
- 不要只说“做了缓存/重构/优化”，要补上结果和影响。
- 不要把所有工作平铺罗列，只挑最关键的 2 到 3 个动作。

## 讲之前的 30 秒检查

- 听众是谁：技术、业务、面试官、老板？
- 对方最关心的是：原理、风险、结果、成本，还是你的个人贡献？
- 我这一段里有没有明确说出：
  `问题`
  `方案`
  `价值`
  `结果`

把下面这段提示词连同你的项目材料一起发给 AI：

```text
请把下面项目材料整理成一个技术人可直接讲出来的项目介绍，输出 3 个版本：
1. 30 秒版本
2. 1 分钟版本
3. 3 分钟版本

要求：
- 先讲结论，再讲细节
- 用“问题-方案-价值”结构
- 明确我的个人贡献
- 尽量保留量化结果
- 不要写营销腔，不要夸大
- 如果原始材料缺少结果、指标或决策原因，请明确标出缺口

项目材料：
[把你的项目描述贴在这里]
```
