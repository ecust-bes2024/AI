# 实验室级 AI 研究与试验模式

## 目的

实验室级规则用于研究、探索和一次性试验。

这层规则的目标不是长期治理，而是低成本验证一个想法值不值得进入正式项目。

## 实验室边界

这里的“实验室”不是单独指某个子目录，而是整个工作区：

- `/Users/jerry_hu/AI`

这个工作区里的典型区域分工如下：

- `lab/`：原型、试验性实现、快速 demo
- `research/`：资料研究、方法吸收、对比分析
- `learn-temp/`：外部仓库、文章、工具、工作流的临时评估区
- `sandbox/`：隔离试跑、冒烟验证、风险较高的试验环境
- `projects/`：从实验结果中逐步升级出来、开始接近正式工程的项目区

因此，实验室级规则首先是工作区运行规则，其次才是单次实验模板。

## 激活条件

当任务满足以下情况时，默认按实验室模式处理：

- 一个想法还没验证过，不确定值不值得做
- 想试一个新框架/工具/agent，但不确定能不能用
- 需要对比两个方案，但还没到正式选型阶段
- 好奇心驱动的探索，目标是"搞清楚这东西行不行"

不进入实验室模式的情况：

- 已有明确需求和稳定落地路径
- 已经决定进入某个正式仓库长期维护
- 已经存在清晰的模块边界、验证流程和资产沉淀计划

这些情况应直接进入项目级。

## 适用范围

- AI 实验室
- 新框架、新 agent、新工具调研
- 提示词、工作流、技能试验
- 一次性 demo
- 架构方向探索

## 核心原则

### 1. 快速验证优先

先回答“值不值得继续”，再回答“怎么做得更完整”。

### 2. 不提前工程化

除非一个实验已经显示出明确复用价值，否则不要过早引入：

- 重型 `AGENTS.md`
- 大型模板目录
- 完整 `module-catalog`
- 大量治理文档

### 3. 先得到升级判断

实验结束时必须给出明确结论：

- `discard` — 方向不对或成本不值，归档
- `keep-as-note` — 有参考价值但不值得继续投入
- `pivot` — 方向对但方案不行，重新设计后开新实验
- `promote-to-project` — 值得升级为正式项目

### 4. 结果必须有流向

实验产出不能只停留在聊天记录里，必须进入以下出口之一：

- `discard`：明确丢弃
- `keep-as-note`：保留为研究结论或方法笔记
- `absorb-to-memory`：吸收为长期偏好、决策或经验
- `promote-to-project`：升级为正式工程资产

## 对个人级规则的放松边界

实验模式下，可以按验证目标适度放松工程完备度，但不应默认降低思考质量。

**通常可以放松的：**
- 可以先做最小可运行实验，而不是完整实现
- 可以接受一次性代码和临时目录结构
- 可以降低文档完备度，只保留关键观察和结论
- 可以先用最快的验证路径，而不是最优雅的工程实现

**仍然保留的底线：**
- 不脑补目标 — 实验的问题和假设必须先写清楚
- 路径不优时必须指出 — 发现当前方案走不通要立刻说
- 时间预算到了必须停 — 不能无限拖延
- 结论必须可判定 — 不能只留下模糊印象

**什么时候不该放松 reasoning：**

- 在比较架构路线时
- 在评估是否吸收外部工具或 repo 时
- 在设计未来可能进入正式项目的工作流时
- 在结论会影响长期方法论或资产沉淀时

## 推荐最小文件

实验室级默认只保留一个轻量文件：

- `experiment-template.md`

如果需要补充，可以再允许：

- `findings.md`
- `comparison.md`
- `todo-next.md`

但默认不要引入完整项目级文档体系。

### findings.md 最小结构

每条发现一行，格式：

```
[日期] 发现内容 → 影响判断（改变了假设 / 确认了假设 / 无关）
```

示例：
```
[2026-04-09] Cloudflare Workers 免费额度每天 10 万次请求 → 确认了假设（免费额度够用）
[2026-04-09] D1 不支持 JOIN 超过 3 张表 → 改变了假设（需要重新设计数据模型）
```

### comparison.md 最小结构

| 对比维度 | 方案 A | 方案 B | 结论 |
|----------|--------|--------|------|
| 维度 1   | ...    | ...    | 选 A/B/都行 |
| 维度 2   | ...    | ...    | 选 A/B/都行 |

底部写一句话总结：选哪个，为什么。

## experiment-template.md 建议结构

每次实验至少记录以下内容：

### 1. 问题

这次要验证什么问题。

### 2. 假设

我当前认为最可能成立的判断是什么。

### 3. 时间预算

这个实验最多花多少时间。超时未得出结论，默认判定为 `discard` 或 `keep-as-note`。

建议范围：
- 快速验证：30 分钟以内
- 标准实验：1-2 小时
- 深度探索：半天（上限）

超过半天还没结论，说明问题定义有问题，应该拆分。

### 4. 参考输入

这次实验基于哪些资料、项目、文章、仓库或工具。

### 5. 最小实验步骤

只列最小可验证路径，不做完整实现。

### 6. Pass/Fail 条件

写出具体的、可判定的通过/失败标准。不要写"成本是否可接受"，要写"单次调用成本 < $0.01"。

示例格式：

- PASS：响应延迟 < 2s，输出准确率 > 80%
- FAIL：需要超过 3 次 prompt 迭代才能稳定输出
- FAIL：依赖的 API 没有免费额度

每个实验至少写 1 条 PASS 条件和 1 条 FAIL 条件。模糊的指标不算。

### 7. 风险

提前写明实验可能误判的地方，例如：

- 只在 toy case 有效
- 依赖外部服务
- 隐藏运维成本
- 与正式工程约束不兼容

### 8. 结论

实验结束后输出四种结论之一：

- `discard` — 归档，不再投入
- `keep-as-note` — 记录发现，暂不行动
- `pivot` — 方向保留，方案重设计，开新实验（附：新实验的问题和假设）
- `promote-to-project` — 升级为正式项目

### 9. 升级建议

如果结论是 `promote-to-project`，必须明确：

- 升级到哪个项目
- 应沉淀成什么资产
- 为什么值得升级

如果结论是 `absorb-to-memory`，也必须明确：

- 要吸收成什么类型
- 为什么它适合作为长期规则或经验
- 它会影响以后什么行为

## 实验室级执行流程

推荐按以下顺序执行：

1. 定义问题和假设
2. 设计最小实验
3. 执行并记录观察结果
4. 给出升级判断
5. 根据结论进入对应出口，而不是停在实验目录里

## 什么时候从实验室级升级到项目级

满足以下条件中的 2 到 3 条，就应考虑升级：

- 同类实验会重复出现
- 实验结果已显示明确复用价值
- 涉及高风险边界
- 以后会进入正式仓库
- 需要多人复用或接手

从实验室级升级后的常见目标包括：

- 正式项目仓库
- 技能
- 工作流
- 长期记忆
- 设计文档

## 什么不该升级

- 只有一次性的好奇心验证
- 结论仍然模糊
- 依赖过重、性价比太低
- 对正式工程没有清晰收益

## 和项目级的边界

实验室级只回答：

- 值不值得做
- 大致怎么做
- 是否值得沉淀
- 沉淀到哪里最合适

项目级才回答：

- 正式仓库里怎么落地
- 具体边界是什么
- 哪些模块必须复用
- 哪些文件允许修改

## 一句话总结

实验室级规则只做一件事：

在整个 `/Users/jerry_hu/AI` 实验室工作区里，用最小成本验证一个方向值不值得被丢弃、记住、吸收，或升级成正式工程资产。
