# 项目级 AI 工程治理方案

## 目的

项目级规则用于正式工程仓库，目标不是”让 AI 多写代码”，而是让 AI 在受控边界内优先复用现成资产，只写必要的胶水代码。

这层规则服务于稳定交付、长期维护和高复用场景。

## 激活条件

当仓库满足以下任意条件时，适合启用项目级治理：

- 仓库已有 3 个以上可复用模块（通信层、UI 骨架、日志等）
- 错误成本高——改错一行可能导致设备通信中断、数据损坏或发布失败
- 多人或多 AI 会话交替操作同一仓库
- 仓库生命周期超过 3 个月，且仍在活跃迭代

反过来，如果仓库只是一次性脚本、短期实验或个人学习项目，留在个人级即可。

## 适用范围

典型适用场景：

- 长期维护的桌面工具
- 芯片工具和设备通信工具
- 有 UI、通信、日志、打包等稳定模块的工程
- 错误成本高、需要持续迭代的项目

## 核心目标

- 优先复用，不默认重写
- 先定义边界，再允许实现
- 把高风险工程知识沉淀成长期资产
- 让 AI 主要负责连接、编排和少量差异化代码

## 核心方法

### 1. 胶水编程

默认假设如下：

- 80% 到 90% 的代码应来自现有模块、模板或样板实现
- AI 主要负责 10% 到 20% 的连接、适配和差异点实现
- 能抄现成资产，就不让 AI 从零创作

### 2. SPEC 与资产分离

- `SPEC` 负责定义这次做什么
- 现成模块、模板和规则负责定义怎么做

两者缺一不可。

## 推荐文件结构

项目级建议最小落地为以下文件和目录：

- `AGENTS.md`
- `spec-template.md`
- `module-catalog.md`
- `templates/`
- `playbooks/`

### 最小启动路径

不需要一次性创建所有文件。分阶段渐进：

1. 第一步：只写 `AGENTS.md`——定义仓库边界和禁区。这一个文件就能让 AI 不乱来。
2. 第二步：下次接到任务时，用 `spec-template.md` 写第一份任务规格。
3. 第三步：每次复用现有模块时，往 `module-catalog.md` 补一条记录。
4. `templates/` 和 `playbooks/` 按需沉淀——遇到重复骨架就提模板，遇到排障超过 3 步就写 playbook。

## 文件职责

AGENTS.md 和 spec-template 的分工原则：AGENTS.md 定义不随任务变化的仓库常量，spec-template 定义随任务变化的变量。

### 1. AGENTS.md — 仓库常量

定义仓库级硬约束，这些规则跨任务不变：

- 技术栈和入口文件
- UI 框架约束
- 线程模型约束
- 通信协议与设备连接约束
- 日志与诊断要求
- 打包与发布要求
- 禁止 AI 直接重写的核心模块（永久禁区）
- 默认测试与验证流程

`AGENTS.md` 回答的问题是：

- 这个仓库的规矩是什么
- 哪些约束对所有任务都生效
- 哪些模块永远不能碰

### 2. spec-template.md — 任务变量

每次任务开始前都先写一份轻量任务规格，定义本次任务的范围：

- 目标——这次做什么
- 现成可复用模块——这次从 module-catalog 里选什么
- 本次允许改动的文件范围（在 AGENTS.md 永久禁区之外的子集）
- 本次的风险点
- 验证方法——这次怎么确认做对了

`spec-template.md` 回答的问题是：

- 这次具体做什么
- 这次应该复用什么
- 这次的边界在哪里

### 3. module-catalog.md

列出仓库里的可复用资产，按类别组织，例如：

- UI 骨架
- logger
- serial / HID / OTA
- task runner / worker thread
- 配置与解析器
- build / package
- 更新与版本管理

每个条目至少记录：

- 模块名
- 路径
- 作用
- 什么时候优先复用
- 什么情况下不要动

`module-catalog.md` 回答的问题是：

- 这个仓库已经有哪些成熟资产
- AI 应该先用哪个，而不是自己新写

### 4. templates/

放可直接套用或改造的标准骨架。

优先沉淀的模板建议包括：

- PySide6 主窗口骨架
- 对话框或页面骨架
- QThread / worker 模式骨架
- 串口通信骨架
- 日志模块骨架
- 打包脚本骨架
- 配置读写骨架

### 5. playbooks/

放排障和验证手册，尤其是高风险边界问题。

什么问题值得写 playbook：满足以下任一条件即可——

- 排查路径超过 3 步，不是看一眼就能定位的
- 曾经误判过根因（比如以为是串口问题，实际是线程问题）
- 涉及跨层边界：硬件 / OS / 框架 / 应用之间的交互

示例（嵌入式工具链项目）：

- UI 卡死排查
- Qt 线程问题排查
- 串口收发异常排查
- 设备热插拔异常排查
- DLL / so / dylib 加载失败排查
- PyInstaller / Nuitka 打包异常排查
- 真机问题复现与日志采集流程

## 项目级任务执行流程

推荐按以下顺序执行：

1. 先写 `spec-template.md`
2. 从 `module-catalog.md` 里选择优先复用模块
3. 判断这次是否需要从 `templates/` 套骨架
4. 如涉及高风险边界，先查对应 `playbooks/`
5. 再开始实现胶水代码
6. 最后按 `AGENTS.md` 的验证要求完成测试和检查

## 什么适合沉淀成项目资产

优先沉淀这些类型：

- 重复出现 3 次以上的功能
- 高风险模块
- 现场问题频发模块
- 打包和发布链路
- 新人最容易写错的模块
- 跨线程、跨进程、跨语言边界模块

## 什么不该沉淀成项目资产

- 一次性实验代码
- 短期临时脚本
- 还没稳定的探索性实现
- 只在单个任务里出现一次的特殊逻辑

## 示例：嵌入式工具链项目的资产优先级

以 PySide6/Qt6 桌面工具 + 设备通信的项目类型为例，资产沉淀优先级建议：

- `serial / hid / ota / i2c` 通信层
- `logger / diagnostics` 诊断层
- `PySide6 / Qt6` UI 骨架层
- `build / package / updater` 发布层
- `binary / struct / config` 解析层

其他项目类型应根据自身的高风险边界和复用频率调整优先级。

## 退出条件

项目级治理不是永久的。以下情况可以降级回个人级或直接移除：

- 仓库不再活跃维护（超过 6 个月无实质性提交）
- 从未产生过可复用资产（module-catalog 始终为空）
- 仓库从正式工程降级为实验/归档状态

## 一句话总结

项目级规则只做一件事：

把高风险、可复用、会重复使用的工程知识沉淀成稳定资产，让 AI 主要做受控的胶水编排。
