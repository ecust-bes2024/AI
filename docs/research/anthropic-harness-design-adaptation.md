# Anthropic Harness Design 吸收稿

**来源**: [Harness Design for Long-Running Apps](https://www.anthropic.com/engineering/harness-design-long-running-apps)
**日期**: 2026-03-29
**目的**: 将文章中的可迁移设计吸收到当前工作方式，而不是复制 Anthropic 的完整实现。

---

## 这篇文章对当前工作方式的意义

这篇文章最值得吸收的不是多 agent 结构本身，而是长任务的控制方法：

- 在编码前先把目标扩成可验证的交付契约
- 把执行和评估拆开，避免 agent 自评过宽
- 在任务跨越多个阶段时主动重置上下文，而不是硬拖长对话
- 随模型能力升级持续删减脚手架，不把 orchestration 当成固定架构

这些原则适合当前工作区常见的三类任务：

- agent / skill / memory / prompt workflow 设计
- 自动化和工具链编排
- 带有前后端联动或多步验收的软件任务

## 应吸收的稳定原则

### 1. 先定义完成，再开始执行

模糊目标先变成可验证的完成标准，再进入实现阶段。

适用信号：

- 需求短但输出复杂
- 用户要的是“成品”，不是代码片段
- 成功标准包含风格、质量、可用性或完整度

落地动作：

- 先写一份短 spec
- 明确 must-have、non-goals、验收检查项
- 如果涉及 UI 或内容质量，补充主观维度评分标准

### 2. 执行和验收分离

同一个 agent 一边生成一边自我背书，可靠性通常不够。

落地动作：

- `builder` 负责产出
- `reviewer` 负责独立找错、找漏项、找偏题
- reviewer 优先用 checklist、测试、快照、lint、运行结果来约束结论

### 3. 长任务不要靠无限上下文硬撑

任务一长，模型容易丢失边界、漂移目标、重复劳动。

落地动作：

- 阶段结束后生成 handoff summary
- 新阶段优先从 summary 开始，而不是继承全部上下文
- 当任务出现反复返工、输出变散、上下文过长时，主动 reset

### 4. Harness 是临时支架，不是信仰

如果模型本身已经能稳定完成，继续堆 planner / evaluator / retries 只会增加复杂度。

落地动作：

- 只有在任务明显超出单 agent 稳定边界时才加 harness
- 新模型、新工具或新流程上线后，重新验证哪些环节仍然需要
- 默认先用最轻版本，再按失败模式加控制层

## 适合当前工作区的轻量 Harness

这是比文章更轻、但更适合当前实际使用的版本。

### Phase 1: Contract

先产出一个简短交付契约，至少包含：

- 目标是什么
- 不做什么
- 输出物是什么
- 如何判断完成
- 需要哪些验证

建议用于：

- 跨文件改动
- 需要测试或验收
- 需要“看起来对”之外的质量保证

### Phase 2: Planner

如果任务复杂度较高，再显式规划。

planner 输出应尽量短，只包含：

- 任务拆分
- 依赖顺序
- 风险点
- 每一步的验收方式

不建议 planner 产出大段空泛分析。

### Phase 3: Builder

builder 只负责交付，不负责定义标准。

builder 的要求：

- 按 contract 实现
- 阶段性汇报已完成和未完成项
- 修改后立即做最小验证
- 如果发现 contract 不完整，先回补 contract 再继续

### Phase 4: Reviewer

reviewer 与 builder 的关注点不同：

- 是否满足 contract
- 是否引入回归
- 是否存在边界条件遗漏
- 是否需要额外测试或手工验证
- 是否出现“看起来完成但证据不足”

reviewer 默认比 builder 更保守。

### Phase 5: Reset or Handoff

出现以下信号时，应该做一次 summary handoff：

- 对话上下文已经很长
- 任务已跨越多个子目标
- 开始重复解释背景
- 返工次数上升
- 新阶段关注点与前一阶段明显不同

handoff summary 建议包含：

- 当前目标
- 已完成内容
- 未解决问题
- 关键文件和状态
- 下一步唯一优先动作

## 什么时候启用这套 Harness

建议启用：

- 长时间、多阶段任务
- 需要独立验收的编码任务
- UI、内容质量、用户体验这类主观要求较多的任务
- 需要先研究再实现的集成或工作流设计任务

不建议启用：

- 单文件小修改
- 一次性脚本
- 结论明显、验证路径短的任务
- 纯信息查询或快速答疑

---

## 当前可直接复用的默认规则

可以作为当前工作方式的默认判断：

1. 任务超过一个实现阶段时，先写 contract。
2. 任务涉及“完成度”而不是“改一行”时，执行和验收分离。
3. 任务开始反复返工时，不继续堆上下文，先做 handoff summary。
4. 任何 harness 组件都要定期删减；保留能明显降低失败率的部分。

---

## 建议的最小落地形式

如果以后要把这篇文章进一步产品化，优先顺序应为：

1. 作为 prompt workflow 使用
2. 作为长期记忆保存 2 到 4 条稳定原则
3. 如反复使用，再升级成可复用 skill

不建议直接把 Anthropic 的完整多 agent 结构固化成默认模板。
